# Public Code Research Library

This is a general-purpose useful-code research collection, not a Curator-only pattern library. It records public repositories, professional website references, reusable components, implementation patterns, design systems, business/product references, and technical findings that may be useful across future projects.

## What Belongs Here

A repository dossier is a human-authored analysis of one upstream repository: provenance, pinned commit, license evidence, useful files, architecture, risks, and recommendations. A reusable asset is a smaller thing extracted from one or more repositories: an algorithm, component pattern, configuration, workflow, UI behavior, or implementation technique.

Verified claims must be traceable to an existing repository file, upstream repository metadata, an official live website, a LICENSE/package metadata file, or an explicitly documented inference. Unverified claims stay marked as `summary-only`, `pending`, or `needs-verification`.

## Structure

| Path | Role |
|---|---|
| `manifest.yaml` | Canonical repository manifest: IDs, upstream URLs, pinned commits, categories, license status, analysis paths, and useful paths. |
| `candidate-pool.yaml` | Canonical candidate status list with controlled status values. |
| `repositories/` | Full repository dossiers. |
| `categories/` | Category-level catalogues and maps. |
| `professional-websites/` | Website and official-demo classification, including exact live-site/source pairs. |
| `synthesis/` | Canonical synthesis reports, rankings, licensing summary, shortlist, and asset catalogue. |
| `rejected/` | Current rejected and needs-verification decisions. |
| `data/raw/` | Raw/generated evidence files kept separate from human-authored reports. |
| `archive/` | Historical or superseded reports retained for provenance only. |

## Status Vocabulary

Candidate statuses use these controlled values: `accepted`, `accepted-reference-only`, `accepted-asset`, `full-dossier`, `summary-only`, `needs-verification`, `rejected`, `superseded`, `duplicate`, and `archived`.

Historical notes may mention earlier classifications, but each current candidate should have exactly one current `status` in `candidate-pool.yaml`.

## Website Classification

Professional website entries are not all the same kind of evidence. Use these categories:

1. Exact verified live-site/source pair
2. Official open-source company website
3. Framework or starter with live demo
4. Design system
5. Product/application demo
6. Personal developer portfolio
7. Professional commercial inspiration only
8. Unverified candidate
9. Rejected or non-public-source example

An exact verified live-site/source pair requires a working live URL, a public source repository, strong evidence that the repository powers that exact site, a verification date, and license status. Frameworks such as Hydrogen or Commerce are not automatically exact sources for real commercial stores; healthcare demo apps are not actual hospital marketing websites.

## License Handling

The active licensing summary is `synthesis/licensing-and-provenance.md`, generated from `manifest.yaml`. A root `LICENSE` file, package metadata, source-file SPDX headers, and custom terms are different evidence types and must not be collapsed into one claim. No-license or unclear-license repositories are study/reference only unless a later pass verifies permission.

## Coverage Limits

Current coverage is still limited for exact public-source matches in several areas: real hospital websites, doctor or clinic websites, commercial product-company websites, real Shopify brand stores, and verified scroll-driven 3D live-site/source pairs. Existing framework demos and starter templates should be treated as references, not proof that they power a specific commercial site.

## Adding Entries

1. Add or update `manifest.yaml` with upstream URL, pinned commit, category, license status, and analysis path.
2. Add a dossier under `repositories/` only when there is actual evidence to support it.
3. If only summary evidence exists, set `analysis_document: null` and `analysis_status: summary-only` or `pending`.
4. Update `candidate-pool.yaml` with one current controlled status.
5. Put raw generated evidence in `data/raw/`; put synthesis in `synthesis/`.
6. Run `python scripts/validation/validate_public_code_library.py` and fix every failure.
